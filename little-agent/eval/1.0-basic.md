# 第一章：提示词基础测试

本章测试模型在最基本条件下的行为稳定性，是所有后续测试的前提。每个用例都是独立 session，单轮或极少轮次。

---

## 1.1 单轮对话（Hello World）

**脚本**：`1.1-basic_hello.py`

**目的**：验证模型能正常接收 prompt 并返回回复。这是所有测试的基础，任何更复杂的用例都依赖此能力。

**设计**：

- 无 system prompt，无工具注册
- 发送一条固定 user prompt：`"What is 1 + 1?"`
- 等待模型回复

**通过条件**：

- 收到非空回复
- 回复中包含 `"2"`（大小写不敏感子串匹配）
- elapsed < 60s

**重复次数**：3 次，每次独立 session。3/3 通过视为通过。

---

## 1.2 工具调用触发

**脚本**：`1.2-basic_tool.py`

**目的**：验证模型能通过 prompt 被诱导调用工具。只要任意工具被调用一次，即视为通过。这是工具相关测试的基础前提。

**设计**：

- 注册 `read` 工具（mock，返回固定内容）
- 注册 `bash` 工具（mock，返回固定内容）
- 发送 prompt：`"What does the file /etc/hosts contain?"`
- 检查 session.messages 中是否有 AssistantNode 包含非空 tool_calls

**通过条件**：至少调用了一次工具（不限 read 或 bash）。

**重复次数**：3 次，3/3 通过视为通过。

---

## 1.3 工具调用意图梯度

**脚本**：`1.3-basic_gradient.py`

**目的**：测试 prompt 的明确程度对工具调用触发率的影响，以及 system prompt 强度能否弥补 prompt 模糊带来的不足。

**设计**：参见 `prompt.md` 第一章 1.1 节（工具调用意图梯度），7 组矩阵（M1–M7），每组 5 次重复，共 35 次调用。

三级 user prompt 清晰度：

- L1（明确）：`"Use the read tool to show me the contents of /etc/hosts."`
- L2（隐含）：`"What does the file /etc/hosts contain?"`
- L3（模糊）：`"How is hostname resolution configured on this system?"`

两级 system prompt：

- SP-weak：`"You have access to tools. Use them when appropriate."`
- SP-strong：`"When users ask about file contents or system configuration, always use the available tools to retrieve accurate information rather than guessing."`

**评判**：记录各组 `call_rate`（5 次中调用工具的次数），不定义 pass/fail，只记录分布。

---

## 1.4 System vs User Prompt 冲突

**脚本**：`1.4-basic_conflict.py`

**目的**：当 system prompt 和 user prompt 存在直接冲突时，模型服从哪一方。覆盖五类典型冲突维度。

**设计**：参见 `prompt.md` 第一章 1.2 节（System vs User Prompt 冲突），5 组，每组 3 次重复，共 15 次调用。无工具注册。

| 组 | 冲突维度 | 自动判断 |
|----|---------|---------|
| C1 | 语言（system 英文 vs user 中文） | chardet 检测输出语言 |
| C2 | 格式（system JSON vs user 纯文本） | json.loads + "answer" key |
| C3 | 长度（system 一句话 vs user 要详细） | 句子数统计 |
| C4 | 范围（system 只聊烹饪 vs user 问巴黎） | "paris" 是否出现 |
| C5 | 语气（system 学术 vs user 幼稚口语） | 人工评估 |

**评判**：C1–C4 自动判断胜出方（system / user / mixed）；C5 输出原文供人工判断。
