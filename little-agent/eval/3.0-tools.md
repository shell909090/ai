# 第三章：工具调用测试

本章测试模型在多工具环境下的工具选择行为：名称、描述、历史上下文三种信号的优先级，以及功能重叠时的选择偏好。

---

## 3.1 工具交换测试（Swap）

**脚本**：`3.1-3.1-tools_swap.py`（即现有 `3.1-3.1-tools_swap.py`）

**目的**：交换一对工具的名称（描述和参数保持原地），观察模型在 baseline（原始）和 swapped（交换后）状态下各选择哪个工具，从而确定名称、描述、历史三种信号的优先级。

**设计**：2³ = 8 组完整因子实验，每组 baseline + swapped 各 3 次，共 **48 次调用**。

三个因素：

| 因素 | 取值 |
|------|------|
| A：工具名称语义熟悉度 | 常见名（bash/read）vs 生僻名（sneave/glorb） |
| B：Prompt 是否显式提及工具名 | 不含工具名 vs 含工具名 |
| C：历史上下文中是否有竞争工具调用 | 无历史 vs 注入 5 轮竞争历史 |

**判断逻辑**（以 bash↔read 为例，swapped 后 bash 持 read 描述，read 持 bash 描述）：
- 调用原名对应工具 → **名称主导**
- 调用描述对应工具 → **描述主导**
- 调用其他工具 → **信号冲突时退化**

**关键结论（step-3.6）**：历史 > 显式命名 ≈ 描述（无历史时显式命名有效）。

## 3.2 工具选择偏好测试（Select）

**脚本**：`3.2-3.2-tools_select.py`（即现有 `3.2-3.2-tools_select.py` 的 T-B 部分）

**目的**：注册多个能完成同一任务的功能重叠工具，prompt 不指定工具名，观察模型在冷启动时的选择偏好。

**设计**：4 个场景 × 3 种历史条件 × 5 次重复 = **60 次调用**。

| 场景 | 可用工具 | Prompt |
|------|---------|--------|
| B1 读文件 | read, bash | "Show me the content of /etc/hosts." |
| B2 下载网页 | webfetch, bash | "Get the HTML content from http://example.com." |
| B3 搜索文件 | glob, grep, bash | "Find all .py files that contain the word 'asyncio'." |
| B4 写文件 | write, bash | "Create a file named test.txt containing 'hello'." |

3 种历史条件：
- 无历史（冷启动）
- 注入 5 轮专用工具历史（如 5 轮 read 调用）
- 注入 5 轮 bash 历史

**评判**：不定义 pass/fail，只记录各工具被选中次数，供分析选择偏好分布。

## 3.3 工具调用能力测试（Progressive Ban）

**脚本**：`3.3-tools_capability.py`（即现有 `3.3-tools_capability.py`）

**目的**：对于不同类型的任务，在当前工具路径被禁用后，模型如何反应——切换到下一条路径、直接作答还是声称无法完成。通过逐步禁令穷举模型对每类任务所掌握的工具路径，观察深度和边界。

**设计**：9 道任务题，每题最多 8 轮渐进禁令，共最多 **72 次调用**（实际因提前结束通常更少）。执行方式为半手动：每轮观察模型使用的命令，识别路径关键词，追加禁令到 system prompt，开启下一轮。

任务覆盖范围：

| 题号 | 任务 | 领域 |
|------|------|------|
| 3.3.1 | 整数乘法 123×456 | 算术 |
| 3.3.2 | 浮点除法 355÷113（10位小数） | 算术 |
| 3.3.3 | 质数判断（7919） | 数论 |
| 3.3.4 | 文本排序（7个单词） | 文本 |
| 3.3.5 | 普通浮点排序 | 文本 |
| 3.3.6 | 字典序陷阱排序（含 1.100、1.20） | 文本 |
| 3.3.7 | JSON 字段提取 | 结构化 |
| 3.3.8 | HTTP GET | 网络 |
| 3.3.9 | HTTP POST | 网络 |

**关键词禁令表（KEYWORD_BANS）**：脚本内置已知命令关键词到禁令指令的映射。遇到未收录的新路径时，脚本打印 `UNKNOWN METHOD` 并退出（exit code 1），操作员将新关键词加入映射后重跑。

**每轮记录**：

| 字段 | 说明 |
|------|------|
| `tool_used` | bash / http / none |
| `command` | bash 的完整命令字符串，或 http 参数 |
| `command_correct` | 命令写法是否正确（语法、选项、边界情况） |
| `result_correct` | 模型给出的最终答案是否正确 |
| `notable_behavior` | 值得记录的具体行为 |

**评判重心**：模型用了什么工具、命令写法是否正确、最终答案是否正确。"尝试了几种工具"是次要维度，不是核心指标。

## 设计说明

**关于随机名工具实验（randname）**

原 `3.2-3.2-tools_select.py` 包含随机名工具可用性验证（T-C）。该实验测试的问题是：工具名为随机字符串时，模型能否正确使用工具输出。

该用例不纳入本章，原因：工具名随机化并非可行的工程方案——若某模型在随机名下工具调用失效，整套基于工具名信号的方案就不成立；若有模型在随机名下仍能正常工作，这也只是该模型的特性而非通用解法。保留 T-A（swap）和 T-B（select）已足够描述工具选择行为。
