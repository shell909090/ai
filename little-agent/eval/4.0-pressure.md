# 第四章：压力稳定性测试

本章测试模型在持续压力下的稳定性边界，测量在何种条件下模型开始崩溃以及崩溃的形式。

## 重要约束：禁止启用 compressor

**所有压力测试脚本必须以 `compressor=None` 构建 AgentCore，不得从配置文件读取 compressor 设置。**

原因：压力测试的目的是观察裸模型在 context 积累下的原始行为边界。一旦启用 compressor，session 会在达到阈值时自动压缩历史，从而改变 input_tokens 的增长曲线，干扰崩溃条件的触发和记录。压缩本身的效果应在第二章（压缩测试）中单独评估，而非在此与压力测试混合。

4.2（数据投喂压力）尤其不能启用 compressor——该测试的核心前提是 context 持续积累直到撑破，compressor 介入会使测试失去意义。

---

## 崩溃判定

以下任一条件满足即判定为崩溃，终止当轮测试：

| 类型 | 判定标准 | 标记 |
|------|---------|------|
| 超时 | 单轮 elapsed ≥ 300s | `timeout` |
| 推理失控 | output_tokens ≥ 20,000 | `runaway` |
| 上下文截断 | finish_reason=length（超出 max_tokens） | `truncated` |
| 乱回答 | 回复未包含任何预期关键词 | `incoherent` |

每次测试重复 **3 次**（独立 session），记录每次首次崩溃的轮次和 input_tokens，报告中位数和极差。

---

## 4.1 持续对话压力

**脚本**：`4.1-pressure_turns.py`

**目的**：在无任何额外数据注入的情况下，纯粹通过持续对话轮次给模型施压，测量模型因轮次积累而崩溃的轮次。无轮次上限，跑到崩溃为止。

**设计**：

- 无工具，无 system prompt
- 每轮发送中性问答（循环使用 10 条固定问题）
- 无轮次上限（MAX_TURNS=999），依赖崩溃条件终止
- 乱回答检测：回复中不包含该问题的任意一个预期答案关键词（如问"What is the capital of Japan?"，回复不含 "Tokyo" / "Japan" / "capital"）

**两种模式**：

| 模式 | User 消息 | 每轮 input 增量（估算） |
|------|-----------|----------------------|
| Mode A（短问题） | 中性知识问答，~10 token | ~50–300 token/轮 |
| Mode B（长内容） | 同问题 + 150 token 背景段落 | ~200–500 token/轮 |

Mode B 用于加速 input 积累，对照 Mode A 区分轮次驱动 vs token 量驱动。

**记录**：每次重复的 `crash_turn`、`crash_input_tokens`、`crash_type`。

## 4.2 数据投喂压力

**脚本**：`4.2-pressure_data.py`（新脚本）

**目的**：每轮向模型投喂大量文本（约 1000–2000 tokens 的小说段落），然后询问**上一轮段落**中的一个具体细节，直到模型无法正确回忆或开始乱答。

这类压力更接近真实应用场景：模型需要在积累大量 context 的同时保持对历史内容的准确记忆。

**设计**：

- 语料：使用英文小说（如《傲慢与偏见》第一章，Project Gutenberg 公版）按段落切分，每段约 1000 tokens
- 轮次结构：
  - 奇数轮：投喂段落 N 的文本（"Here is a passage: [text]. Read it carefully."）
  - 偶数轮：询问段落 N-1 中的具体细节（如人名、地点、具体行为），同时投喂段落 N+1
- 乱回答检测：回复中不含问题对应的正确答案关键词（每个问题预设 2–3 个可接受答案关键词）
- 无轮次上限

**记录**：每次重复的首次错误回答（`incoherent`）的轮次和 input_tokens。与 4.1 对比，input 增量更大，预期更早触及 token 压力边界。

**语料要求**：
- 至少 30 段，每段约 1000 tokens
- 每段预设一个细节问题 + 答案关键词列表
- 语料文件：`pressure_corpus.jsonl`（不入 git，通过辅助脚本生成）

## 4.3 工具调用数据压力

**脚本**：`4.3-pressure_tools.py`

**目的**：每轮通过工具调用返回大量数据，观察工具返回内容的持续积累是否比纯对话更快导致崩溃。与 4.1 对比，量化工具数据积累的额外压力。

**设计**：

- 注册 `read` 工具（mock）
- 每轮新建 session（避免历史偏向干扰工具调用意愿）
- 同时注册 `bash` 工具（双工具环境触发工具调用，参照 prompt_eval 的有效配置）
- Prompt：`"What does /etc/hosts contain? Use the read tool."` 等文件读取问题
- Mock 工具返回：T-small（~10 token）和 T-large（~200 token）两种变体
- 乱回答检测：回复中不含 `/etc/hosts` 相关内容关键词

**注意**：若模型不调用工具，该轮记为 `no_tool_call`（不算崩溃，但说明工具压力无法施加，视为测试设计对该模型无效）。

**记录**：`crash_turn`、`crash_input_tokens`、`crash_type`、`tool_call_rate`（调用工具的轮次比例）。
