# 流程规范

1. 确认git有没有未提交内容，包括stage区和本地文件。如有，需要用户明确，提交还是忽略。
2. 用户提出需求，或读取用户需求文档取得需求。通过git确认需求是初次提出还是变更。如果有不明确的，要求用户澄清和补充。用户在对话里口头提出需求的，需要确认要不要补充到req.md。
3. 根据用户需求重新审视设计。如果设计变化，修改设计文档design.md。文档应当只包含设计要点，例如数据库结构，重要接口定义，重点业务逻辑等。不需要列出所有细节，但是接口定义需要完整精确。需求设计完成后停一下，等待用户确认。
4. 写代码实现需求。如果需要更新测试代码的，写测试代码。代码编写尽量并行，例如前端和后端。在设计文档定义了接口隔离的基础上，多个组件应当分成多个subagent来实现。业务代码和测试代码，只要可以，就必须用两个subagent分别各自实现。如果解耦有困难，可以先写出函数签名，然后由subagent分别实现代码和测试代码。所有代码均需要遵循质量规范。
5. 运行代码质量检查工具。每次修改代码，包括修改测试代码，都务必运行代码质量检查工具。
6. 运行测试。如有问题可以重新修改设计或修改代码。但不要修改测试回避问题。一切以需求和设计为出发点。每次修改代码都需要重复测试流程。
7. review代码修改，确认逻辑正确。检查是否有未实现功能，是否有安全问题，是否有调试代码未清理，是否有代码质量问题。如果有，把问题按照优先级更新到review.md。然后等待用户检查。用户可以标记某些问题为“暂不修复”，“不是问题”，或补充自己的问题。
8. 读修改后的review.md。对每个问题，依次把“修复问题”作为需求，走流程2-6。修复完成后在review.md中标记为已修复。如此循环，直到问题全部解决。
9. 更新README.md和README.cn.md。里面需要包含项目简介，安装和运维说明，使用说明，作者，版权声明和授权协议。其中作者，版权声明和授权协议不要瞎编。
10. 部署和运行项目进行测试。检查输出日志是否正常。如果可以驱动测试工具，例如驱动浏览器，则驱动浏览器简单使用项目。如果行为和预期不一致，检查日志，修复问题。
11. 整理本轮文档。例如删除review.md中已修复的问题。添加git stage并提交。把本轮重点过程追加到log.md。log.md中需要包含本轮用户需求，测试发现和修复了什么问题，覆盖率多少。review中发现和修复了什么问题。最后总结，是否完成了用户需求。

# 文件规范

* AGENTS.md：项目规范性文档，说明项目怎么做，哪个文件在哪里。AI禁止修改。
* README.md：项目说明。由AI管理，写给人类阅读。英语书写。英文说明中包含对所有其他语言说明的链接。
* README.cn.md：项目说明中文版。
* .env: 项目启动所需环境变量，禁止加入git。
* docs：
   * req.md：需求文档。用户写给AI看的。AI未得用户明确指示不得修改。
   * design.md：设计文档。AI写给系统维护者看的。中文书写。
   * review.md：评审文档。AI和用户共同管理，不需要加入git。中文书写。
   * log.md：AI运行过程中，如果遇到需要记录下来的大事，追加到log.md。完全AI管理，不需要加入git，中文书写，记录需要带时间。

# 代码规范

* 代码，注释，使用英文书写。一般文档使用中文书写，但README.md除外。
* git comment使用英文书写，需要简明扼要，禁止把ai列为co-author。
* 流程使用Makefile控制。
* 代码质量检测工具包括make的fmt，lint，build环节。
* 代码测试包括make的unittest，test环节。
* 测试覆盖率底线是50%，一般是75%。覆盖率低于50%需要警告用户。低于75%需要尽力提升。
* 测试用例需要严格按照文档来，不要受实现影响。
* 代码关键处可以添加注释方便理解，但不要机械描述代码行为。
* 程序的logging系统需要支持调试模式开关。调试信息用调试级别输出。
* 如果可以的话，部署工具尽量使用docker。

# 前端规范

* 优先使用typescript。
* 优先使用 ES6+ 语法。

# python规范

* 强制执行 Type Annotations。
* 公有函数应包含简洁的 Docstrings。不超过一行，注明函数核心功能。
* 环境使用uv管理。
* 使用ruff进行静态检查。其中配置McCabe复杂度阈值为 10。这部分在make的lint环节。
* 删除无用代码，删除头部无效import。
* 使用logging处理日志。
* 使用py_compile检查代码语法。这部分在make的build环节。

# golang规范

* golang代码，文件必须用gofmt格式化，导入分组用goimports，这部分在make的fmt环节。
* golang代码，提交前需要通过go test和golangci-lint，这两项分别在make的test环节和lint环节。
